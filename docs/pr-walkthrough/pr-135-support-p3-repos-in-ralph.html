Why This PR Exists

Ralph is the AI agent supervisor that picks up implementation beads and dispatches Codex workers, but it only knew about two repositories. Phase 3 of the hosted MCP server project (AIML-767) introduces deployment and gateway work that spans four repos: mcp-contrast, aiml-services, aiml-deployment, and contrast-api-gateway. Without this change, Ralph cannot autonomously select or execute beads labeled repo:aiml-deployment or repo:contrast-api-gateway, which means the Phase 3 child beads for Helm deployment, CAG routing, and network policy hardening would all require manual human intervention to start.

To understand why four repos matter, consider what Phase 3 is building. The goal is to deploy a production-ready hosted MCP server on AWS EKS behind the Contrast API Gateway (CAG). That involves:

• aiml-services builds the container image with Jib (bead mcp-18eh, now closed).
• aiml-deployment holds the Helm values, network policies, and HPA config (beads mcp-xn8i.1, mcp-r07w, mcp-y5to.1).
• contrast-api-gateway adds the CAG route for POST /mcp (bead mcp-xn8i.2).
• mcp-contrast is the control repo where beads live and Ralph runs.

Why this matters Without this PR, every Phase 3 bead in aiml-deployment or contrast-api-gateway would fail Ralph's validation gate. Ralph checks that each bead has exactly one repo:* label and that the label maps to a known repository path. A bead labeled repo:aiml-deployment would hit the die "unsupported repo label" error and stop the entire automation loop. This PR adds two repo path mappings and replaces hardcoded validation with a data-driven approach.

This PR is part of a stacked branch chain. It sits on top of PR #134 (AIML-766: Slice 12, full tool parity and release hardening), which itself is stacked on AIML-765 (Slice 11, isolation and load tests). The AIML-767 epic (bead mcp-4s5y) is the third major phase of the hosted MCP server rollout. Phase 1 and 2 proved the architecture and achieved tool parity. Phase 3 deploys it for real.

Design Approach

The design principle is: make the repo list data-driven instead of hardcoded. Before this PR, both scripts used inline case statements and explicit process_repo calls for each repo. Adding a third or fourth repo meant editing multiple places and hoping you got them all. The fix introduces two data structures:

• A SUPPORTED_REPO_LABELS array in ralph with a helper function is_supported_repo_label() for validation.
• A REPOS array in ralph-next-bead that pairs each repo path with its name, iterated with a single loop.

This means adding a fifth repo in the future would be a one-line change in each script, rather than hunting through case branches and function calls. The existing repo_path_for_label() function retains its case statement because it maps labels to paths (a lookup, not a validation), but it gains two new entries.

Note The prompt template also gets a small improvement. Previously, the Codex worker prompt only mentioned reading CLAUDE.md from aiml-services. Now it instructs Codex to read CLAUDE.md or AGENTS.md from whichever implementation repo the bead targets. This matters because aiml-deployment and contrast-api-gateway may use AGENTS.md instead of CLAUDE.md for their coding rules.

scripts/ralph-next-bead

The ralph-next-bead script handles the transition between completed beads: creating PRs, generating walkthroughs, and setting up stacked branches. It processes every repo that has commits on the current branch. Before this PR, it hardcoded two process_repo calls. Now it iterates over a REPOS array, following the same data-driven pattern.

Each entry in REPOS is a pipe-delimited pair: filesystem path and display name. The pipe delimiter is safe here because paths never contain |. The ${repo_entry%%|*} and ${repo_entry##*|} parameter expansions split each entry without spawning a subshell.

The loop replacement is the most visible structural change. Two explicit calls become a three-line loop that processes every entry in the array.

Summary and Review Guide

This is a small, self-contained automation change that prepares Ralph for Phase 3's four-repo scope. No Java code, no deployment config, no test changes. The only risk is a typo in a repo path, which would surface immediately when Ralph tries to dispatch a bead to a nonexistent directory.

Review focus areas Check the following:
1. Are the two new repo paths correct? (/Users/chrisedwards/projects/contrast/aiml-deployment and /Users/chrisedwards/projects/contrast/contrast-api-gateway)
2. Does the SUPPORTED_REPO_LABELS array match the bead labels used in Phase 3? (The epic lists repo:aiml-deployment and repo:contrast-api-gateway, which match.)
3. Is the prompt template improvement reasonable? (Reading AGENTS.md in addition to CLAUDE.md.)
