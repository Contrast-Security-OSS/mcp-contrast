Approach: Three Independent Fixes

Each of the three commits solves an independent problem discovered while preparing S12's child beads for execution. They're bundled in one PR because they're all small, all on the same stacked branch, and all need to land before the S12 child beads can safely proceed. None of the three changes affect runtime behavior of the MCP server itself.

Here's what each fix addresses:

1. -parameters compiler flag — Without this flag, Java strips method parameter names during compilation, replacing them with arg0, arg1, etc. Spring AI MCP uses these names to generate tool parameter schemas exposed to AI agents. With the flag missing, the schemas would show meaningless names.

2. Ralph human-only bead filtering — Ralph is the Bash supervisor script that drives automated bead implementation using Codex. S12 introduces two new bead label categories (human-security-review and external-approval) that Ralph must skip. Without this fix, Ralph would attempt to run security review beads that require a human.

3. PR #131 walkthrough restore — The HTML walkthrough for PR #131 (cursor pagination core abstraction) was accidentally deleted during a prior branch's rebase. This commit restores the 1,514-line file from the commit history.

Parameter Metadata: One Line, Big Impact

Java's compiler discards method parameter names by default — a design choice from 1996 that creates a real problem for modern tool frameworks. When you write public String searchVulnerabilities(String appId, String severity), the compiled bytecode only sees arg0 and arg1 unless you tell javac to preserve the names. Spring AI MCP uses Java reflection to auto-generate JSON schemas for tool parameters. Without the real names, the schema an AI agent sees looks like this:

Without -parameters (what AI agents see) BAD 
 {
   "name": "search_vulnerabilities",
   "parameters": {
-    "arg0": { "type": "string" },
-    "arg1": { "type": "string" }
   }
 }
With -parameters (what AI agents see) GOOD 
 {
   "name": "search_vulnerabilities",
   "parameters": {
+    "appId": { "type": "string" },
+    "severity": { "type": "string" }
   }
 }

The fix is a single line in the root build.gradle. It applies to all subprojects (both contrast-mcp-core and contrast-mcp-stdio-app), ensuring every compiled class carries full parameter metadata.

build.gradle MODIFIED 
 63    tasks.withType(JavaCompile).configureEach {
 64        options.encoding = 'UTF-8'
 65        options.release = 21
+66        options.compilerArgs.add('-parameters')
 67    }
The -parameters flag tells javac to store formal parameter names in the bytecode's MethodParameters attribute. This is the same flag Spring Boot projects commonly enable. It has zero runtime cost — slightly larger class files but no performance impact. The flag is placed in the subprojects block so it applies to both contrast-mcp-core (the published library) and contrast-mcp-stdio-app (the local server).
Why this matters for S12 The S12 child bead mcp-stcg runs the generated schema gate, which snapshots the tools/list response and diffs it against a committed baseline. Without -parameters, the snapshot would contain arg0/arg1 parameter names, and either the gate would fail or the baseline would be wrong. This one-line fix ensures the schema gate sees real parameter names from the start.

Ralph Hardening: Teaching the Automation to Stay in Its Lane

Ralph is the Bash supervisor that runs Codex workers against beads — think of it as the robot foreman on the AI implementation assembly line. When Ralph scans for work, it queries br ready to find unblocked beads, then filters out ones that require human judgment. Before this PR, Ralph knew about two human-only labels: ready-for-human and needs-human-review. But S12 introduces two new categories:

• human-security-review — Beads that require a human security reviewer (like the privacy/data-flow evidence bead mcp-lkk6)
• external-approval — Beads waiting on an external team's sign-off (like the DCR monitoring bead mcp-82qj which depends on the OAuth team)

Without this fix, Ralph would pick up a security review bead and try to implement it — which at best wastes compute and at worst produces incorrect evidence.

Design decision Rather than duplicating the label list in four places (as the old code did), this commit extracts the filter into a single human_only_label_filter jq variable. The variable is interpolated into each call site using Bash string substitution. This means future label additions only need to change one place.

Here's the extracted filter variable — the single source of truth for what Ralph skips:

scripts/ralph MODIFIED 
 112  (cd "$CONTROL_REPO" && BEADS_DIR="$BEADS_DIR" br "$@")
 113}
 114
+115human_only_label_filter='
+116  any(
+117    . == "ready-for-human"
+118    or . == "needs-human-review"
+119    or . == "human-security-review"
+120    or . == "external-approval"
+121  )
+122'
This jq fragment is defined once as a Bash variable and interpolated into four call sites via '"$human_only_label_filter"'. The any() jq function returns true if any element in the input array matches one of the four conditions. The single-quote Bash string ensures no shell expansion happens inside the jq code.
