The Problem: Two Incompatible Pagination Models

The dataplatform observation APIs use cursor/keyset pagination, but the MCP server's existing PaginatedTool only knows about offset-based page numbers. To understand why this matters, imagine an AI agent asking "show me the next page of issue observations." With offset pagination, the agent says page=2 and the server calculates offset = (2 - 1) × 50 = 50. Simple arithmetic. But the dataplatform's observation endpoints don't work this way — they return an opaque continuation token (a cursor) that encodes the position in the result set. The agent must pass that token back verbatim to get the next page.

These are fundamentally different contracts, not just different parameter names. Offset pagination exposes page, pageSize, totalPages, and totalItems — the agent can jump to any page. Cursor pagination exposes cursor, pageSize, nextCursor, and hasMore — the agent can only go forward. Trying to shoehorn cursor semantics into the existing PaginatedTool would mean either:

Option A: Force-fit into PaginatedTool
Expose page and totalPages that don't actually work
Fake page=1 on every request, breaking agent expectations
Hide cursor tokens in response metadata, confusing the AI
Option B: Parallel type hierarchy (chosen)
Clean contract: cursor/pageSize in, nextCursor/hasMore out
No random-access fields that would mislead the agent
Cursor values treated as opaque — never parsed or logged
Security consideration Cursor values can contain backend search mechanics. The dataplatform's older sortAfter mechanism embeds typed field values. Logging, parsing, or exposing these in error messages would leak backend internals to AI agents. The PRD's CursorPaginationContract invariant explicitly requires that cursors are treated as opaque continuation tokens throughout.

There's also a gate to satisfy. The PRD defines a "cursor-client gate" that says list_issue_observations and list_incident_observations cannot be registered unless the client can actually pass an opaque cursor and receive nextCursor/hasMore back. First-page-only implementations are explicitly forbidden. This PR builds the foundation that makes it possible to satisfy that gate.

3

Approach: Mirror, Don't Merge

The design creates a parallel type hierarchy alongside the existing offset pagination types. Rather than trying to generalize PaginatedTool (which would require every existing offset tool to handle nullable cursor fields), the PR adds four new types that mirror the structure of their offset counterparts while enforcing cursor semantics. Here's how the types map:

Offset vs. Cursor type hierarchy

Both hierarchies inherit from BaseTool, sharing the auth strategy and error mapping. The four new cursor types are deliberately shaped like their offset counterparts — same generic patterns, same final executePipeline() template method, same exception-to-user-message mapping — so a reviewer who knows PaginatedTool will instantly recognize the structure. The key differences are:

• CursorPaginationParams accepts cursor + pageSize instead of page + pageSize. No offset, no page field.
• CursorExecutionResult carries nextCursor + hasMore instead of totalItems. No server-side count.
• CursorToolResponse exposes nextCursor + hasMore to the AI agent. No page, totalPages, or totalItems.
• CursorPaginatedTool logs cursorPresence() (“present” or “absent”) instead of the actual cursor value.

CursorPaginatedTool — The Pipeline

The heart of this PR is the executePipeline() method — a final template method that enforces a consistent processing sequence. If you've reviewed any of the existing tools (from PR #124 onward), you'll recognize this pattern from PaginatedTool. The stages are identical: validate → authenticate → execute → respond. But the cursor version adds cursor-specific logging and never exposes the cursor value in any error path.

Test Coverage: 22 Tests Across 3 Classes

Every test in this PR checks two things: the functional behavior, and that cursor values don't leak into errors or warnings. The test classes use a constant OPAQUE_CURSOR = "opaque.cursor/with+symbols==" — a deliberately gnarly string that would be conspicuous if it appeared in an error message. Every test that passes this cursor also asserts the cursor value is absent from the response.

Why this pattern matters The opaque cursor test constant is intentionally ugly. If any code path accidentally includes the cursor in a log message, error string, or warning, the assertion noneMatch(msg -> msg.contains(OPAQUE_CURSOR)) catches it. This dual-assertion pattern (functional correctness + leak detection) runs on every single test.

Diagnostic Gate Script

The script hack/verify-s9d-cursor-pagination.sh is a comprehensive diagnostic gate that validates both runtime behavior and source-level invariants. It runs the focused cursor tests, scans the Gradle output for forbidden secrets/cursor values, then performs static assertions on the source files. This is the same pattern used by previous slice gates (S4A, S4B, S4C, S8).
