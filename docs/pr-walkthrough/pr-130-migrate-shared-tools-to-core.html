Context: The Road to Tool Sharing

Contrast Security's MCP server lets AI agents query vulnerability data, but until now every tool was locked inside the local stdio application — useless to the hosted remote server being built in parallel. The hosted server needs these exact same tools, but authenticates differently: bearer tokens via an API gateway instead of local API keys. Slice 8 is the PR that liberates these tools from their local prison and places them in a shared contrast-mcp-core module that both servers can consume.

Think of it like moving tools from a toolbox bolted to one workbench into a shared pegboard that any workbench can reach. The tools themselves don't change what they do — search_vulnerabilities still searches vulnerabilities, get_route_coverage still retrieves route coverage. What changes is how they talk to the Contrast API. Instead of reaching directly into the SDK (which requires local API keys), they now call through a ContrastApiClient interface that can be backed by either the local SDK or a remote bearer-token HTTP client.

Why this matters This is the biggest single PR in the AIML-110 initiative. It migrates 12 of the 14 tools that will be shared between local and hosted servers. After this, hosted server users gain access to the full TeamServer security tool set — vulnerability search, application discovery, attack analysis, library SCA, route coverage, and SAST metadata — once the hosted side wires up its BearerTokenApiClient implementation.

This PR sits in the middle of a 12-slice implementation plan. Earlier slices built the foundation; later slices will expand to dataplatform tools, staging validation, and release hardening. Here's where we are:

The Migration Pattern

Every tool migration follows the same four-step formula. Once you understand this pattern from the first migration (Protect Rules in Chapter 3), the remaining 11 tool migrations become predictable. Here's what changes when a tool moves from contrast-mcp-stdio-app to contrast-mcp-core:

Before and After: How a Tool's Dependencies Change

The four steps of every migration:

1. Rename: Move the tool class and its params class from contrast-mcp-stdio-app/src/main/java/.../tool/{family}/ to contrast-mcp-core/src/main/java/.../tool/{family}/. The package stays the same — only the Gradle module changes.

2. Rebase: Change the base class from LocalSdkSingleTool or LocalSdkPaginatedTool (which inject SDK factories) to the shared SingleTool or PaginatedTool (which have no SDK dependency).

3. Inject: Replace direct SDK calls like getContrastSDK().getProtectConfig(getOrgId(), appId) with contrastApiClient.getProtectRules(appId). Notice how orgId disappears from the call — the ContrastApiClient implementation resolves it internally (from an env var locally, or from the JWT token remotely).

4. Forward: Add ToolContext toolContext as the last parameter of the @Tool method and pass it to executePipeline(..., toolContext). This enables the base class to invoke the AuthenticationStrategy on the hosted path, binding the caller's bearer token before any API call is made.

Concrete example Before: extendedSDK.getProtectConfig(getOrgId(), params.appId()) — the tool knows it's talking to the SDK, knows the org ID, and is locked to local API key auth.
After: contrastApiClient.getProtectRules(params.appId()) — the tool just says "get me protect rules for this app" and the injected implementation handles the rest. On the hosted server, that implementation would send an HTTP request to the Contrast API Gateway with the user's bearer token.

The mass file movement is the most visually dramatic part of this PR. 38 Java files were renamed (moved between modules). Of these, 17 were pure renames with 100% content similarity — the file literally didn't change, just its home. The other 21 had content modifications (the rebase/inject/forward steps above). GetProtectRulesTool was the first tool migrated because it's the simplest — a single API call, no pagination, no enrichment. It retrieves Contrast Protect's runtime protection rules for an application. The migration exemplifies every step of the pattern in a small, reviewable package. Every subsequent migration in this PR follows the same formula, so understanding this one means understanding all 12.

Vulnerability Tools: The Complex Family

GetVulnerabilityTool is the most complex tool in the entire MCP server. It doesn't just fetch one thing — it orchestrates multiple API calls to assemble a rich vulnerability report. After fetching the core vulnerability trace (with expanded application, environment, and session data), it makes up to five additional calls for recommendations, HTTP request data, event summaries, library data, and library vulnerability observations. Each secondary call uses graceful degradation via WarningCollector — if one enrichment call fails, the tool still returns the core data with a warning instead of failing entirely.

GetVulnerabilityTool: Orchestrated API Calls

The migration preserved this entire enrichment chain but routed every call through ContrastApiClient. Where the old code called sdk.getRecommendation(orgId, vulnId), sdk.getHttpRequest(orgId, vulnId), and sdkExtension.getEventSummary(orgId, vulnId), the new code calls contrastApiClient.getRecommendation(vulnId), contrastApiClient.getHttpRequest(vulnId), and contrastApiClient.getEventSummary(vulnId).

Security consideration The library enrichment chain is the highest-fanout path in the tool set. getAllLibraries(appId) can return hundreds of libraries, and for each library that matches a stack frame, getLibraryObservations() makes another API call. On the hosted path, every one of these calls goes through the API Gateway with the user's bearer token — no caching is permitted per the NoRemoteStaticCache invariant. The PRD mandates observe-first telemetry (not preemptive truncation) for these high-fan-out paths.
SearchVulnerabilitiesTool + SearchAppVulnerabilitiesTool — Standard Migration Standard Pattern Follows: GetProtectRulesTool (PaginatedTool variant)
Both are paginated search tools that follow the exact same migration pattern as the protect rules tool, but extending PaginatedTool instead of SingleTool. The only difference is SearchAppVulnerabilitiesTool takes an additional appId parameter to scope searches to a single application. SDK calls like sdk.getTracesInOrg(orgId, traceFilterBody) became contrastApiClient.searchVulnerabilities(traceFilterBody).

Libraries, Attacks, Coverage, and SAST

The remaining four tool families follow the established migration pattern with minor family-specific variations. Each family was migrated in its own commit for incremental verification. The variations are noted below, but the structural changes are identical to the protect rules exemplar.

Notable variation: Library tools ListApplicationsByCveTool has a secondary enrichment loop similar to GetVulnerabilityTool. After fetching applications affected by a CVE, it calls contrastApiClient.getAllLibraries(appId) for each application to match vulnerable libraries. The old code used SDKHelper.getAllLibraries() which could hit a local Guava cache. The new code goes through ContrastApiClient — on the hosted path this means uncached calls through the API Gateway, per the NoRemoteStaticCache invariant.
Notable variation: Library params ListApplicationLibrariesParams and ListApplicationsByCveParams had direct SDK type references removed. The params classes previously imported SDK-specific types for validation; these were replaced with string-based validation that works without SDK on the classpath.

Changed from extendedSDK.getAttacks(getOrgId(), filterBody, limit, offset, sort) to contrastApiClient.searchAttacks(filterBody, limit, offset, sort). The 401 Bug Fix

PR #129 (Slice 7) inadvertently changed the 401 error message to "Your authentication token has expired" — which is wrong for two reasons. First, the local stdio server uses API keys, not tokens. API keys don't expire, so telling users their "token expired" sends them down the wrong troubleshooting path. Second, the Contrast API intentionally returns 401 for unknown resource IDs as a security measure (to avoid leaking whether a resource exists). So a 401 doesn't always mean "auth failed" — it can also mean "that vulnerability/application/attack ID doesn't exist."

tool/base/BaseTool.java — mapHttpErrorCode() MODIFIED 
 62  protected String mapHttpErrorCode(int code) {
 63    return switch (code) {
-64      case 401 -> "Your authentication token has expired. Please re-authenticate and retry.";
+64      case 401 ->
+65          "Authentication failed or resource not found. Verify credentials and that the"
+66              + " resource ID is correct.";
 67      case 403 -> "Access denied. User lacks permission for this resource.";
 68      case 404 -> "Resource not found.";
The restored message deliberately covers both failure modes: bad credentials and unknown resource IDs. This is the message that was in place before PR #129 changed it. Three integration tests were failing because they send bogus IDs to verify the error path, and the API returns 401 (not 404) for those IDs. The tests expect the dual-purpose message.

The same commit also tightened log messages in PaginatedTool and SingleTool. Error log lines previously included e.getMessage() in the log message, which could leak downstream error details. These were changed to fixed strings ("Request failed" and "API error") with the exception type and HTTP status preserved as structured key-value pairs via SLF4J fluent logging.

tool/base/SingleTool.java & PaginatedTool.java MODIFIED 
     log.atWarn()
         .addKeyValue(LoggingKeys.REQUEST_ID, requestId)
         .addKeyValue(LoggingKeys.EXCEPTION_TYPE, e.getClass().getSimpleName())
-        .setMessage("Request failed: {}")
-        .addArgument(e.getMessage())
+        .setMessage("Request failed")

Removing e.getMessage() from log messages prevents downstream error details from leaking into logs. The exception type (IOException, UnauthorizedException) is preserved as a structured key-value for diagnostics, but the potentially sensitive message content (which might include URLs, org IDs, or error codes from the Contrast API) is no longer logged. This aligns with the Slice 7 credential redaction policy.

Testing Strategy

The migration uses a three-tier testing approach that creates a ratchet: once a tool is migrated, tests at every level prevent regression.

Three-Tier Testing Architecture

Tier 1 core tests verify the tool's logic in isolation from any backend. They mock ContrastApiClient and assert that the tool routes calls correctly, validates inputs, forwards ToolContext, preserves its @Tool name, generates warnings for empty data, and sanitizes exceptions into request-ID-tagged error messages. These tests live in contrast-mcp-core alongside the tool, so they ship in the published artifact.

Tier 2 local parity tests verify that SdkApiClient correctly bridges the tool to the local SDK. They mock the Contrast SDK and SDKExtension, wire up a real SdkApiClient and real tool, then assert the full chain produces correct results. Critically, they also assert that the local organization ID does not leak into the tool's response JSON — ensuring the response is transport-agnostic.

Tier 3 integration tests already existed for most tools and were updated to reflect the corrected 401 message. They run against a real Contrast TeamServer with seeded data and verify that filters, pagination, sort, and error paths all work end-to-end.

Verification Scripts (7 new shell scripts) Standard Pattern hack/verify-s8-*.sh Follows: hack/verify-s4a-client-boundary.sh
Seven per-family verification scripts follow the same structure as the Slice 4 boundary scripts. Each script runs make check-test with filters for the relevant test classes, checks exit codes, and emits structured assertions. They serve as quick sanity checks during development and as CI-reproducible evidence that each family passes independently.
Deleted stdio-app tests (4 files, replaced by core tests) Standard Pattern contrast-mcp-stdio-app/.../*ToolTest.java Replaced by: contrast-mcp-core/.../*ToolTest.java
Four test files were deleted from contrast-mcp-stdio-app: GetProtectRulesToolTest, SearchAttacksToolTest, GetRouteCoverageToolTest, and GetSastProjectToolTest. These were not simply moved — they were replaced by new core test classes with updated assertions (testing against ContrastApiClient mocks instead of SDK mocks). The core tests provide equal or greater coverage. Local parity tests in stdio-app cover the remaining local-specific concerns.
