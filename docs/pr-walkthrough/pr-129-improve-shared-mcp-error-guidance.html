Context: Why Error Messages Matter for AI Agents

MCP tools are consumed by AI agents, not humans — and AI agents can only recover from errors they understand. Imagine Claude calling search_vulnerabilities and getting back "Contrast API error. Try again later." That message tells the agent nothing. Try again how? Is the query too broad? Are credentials expired? Should the agent narrow filters or re-authenticate? Now imagine getting "The service returned an error. Narrow filters or reduce page size, then retry." — that's an instruction the agent can act on autonomously.

This PR is part of Slice 7 (Error and Logging Hardening) in the AIML-110 initiative to build a hosted remote MCP server. The PRD defines two stop-ship gates that depend on these shared error messages: S7-ERROR-LOGGING requires layer-specific error handling per ADR-007, and G2-ERRORS mandates that every error response be LLM-readable with no stack traces, class names, or internal paths. The exact wording for specific status codes is specified in the PRD — for example, a JWT mid-execution expiry must map to "Your authentication token has expired. Please re-authenticate and retry."

Why this matters This is the shared foundation. BaseTool.mapHttpErrorCode lives in :contrast-mcp-core, which is consumed by both the local stdio server and the hosted remote server. Fixing the messages here automatically improves every tool in both deployment modes — all 13 local tools and all 12+ remote tools being added in Slices 8 and 9.

Approach: Before and After

The change is deliberately narrow: rewrite five error messages and eliminate duplicated strings. The PRD prescribes exact wording for each HTTP status code. Rather than inventing messages, we're implementing the specification. Along the way, we found and fixed a bug: 403 responses were being caught as UnauthorizedException instead of HttpResponseException, which meant they got the 401 message instead of "Access denied."

The Core Change: BaseTool.mapHttpErrorCode

This is the single method that every MCP tool in both servers calls when an HTTP error comes back from the Contrast API. The switch expression maps status codes to the messages that AI agents see. Every message must give the agent a concrete next step.

Three messages changed, each with a specific rationale. 401: The old message conflated two unrelated problems — bad credentials and wrong resource IDs. The PRD (G2-ERRORS gate) requires this exact wording for JWT mid-execution expiry so the agent knows to re-authenticate, not retry the same request. 429: "Retry later" is vague; "retry after a brief pause" tells the agent the wait is short, not hours. 5xx: "Contrast API error" leaks the vendor name and gives no recovery path. The new message gives the agent two concrete things to try: narrow the query or reduce page size — both common causes of server-side timeouts on large result sets.
Note The 403 and 404 messages were already correct and remain unchanged. The default fallback also stays — it includes the HTTP code for debugging, which is appropriate for unexpected status codes.

Eliminating Duplicated Error Strings

Before this PR, both PaginatedTool and SingleTool hardcoded the 401 error message in their UnauthorizedException catch blocks. That meant three copies of the same string: one in BaseTool.mapHttpErrorCode and one in each subclass. If you updated the message in BaseTool but forgot the subclasses, the agent would see different messages for the same error depending on whether the SDK threw UnauthorizedException directly or it came through HttpResponseException routing. The fix is simple: delegate to mapHttpErrorCode(401).

The 403 Bug: Wrong Exception Type

The existing 403 test cases were throwing UnauthorizedException for a 403 status — which masked a real routing bug in production. Here's the problem: the Contrast SDK's UnauthorizedException extends HttpResponseException. When the SDK returns a 403, it throws HttpResponseException with status code 403, not UnauthorizedException. UnauthorizedException is only thrown for 401 responses. But the old tests constructed an UnauthorizedException with a 403 status code — an impossible combination in production — which meant the catch block matched and the test passed with the wrong message.

Security consideration A 403 (forbidden) and a 401 (unauthorized) require fundamentally different agent responses. A 401 means "your token expired, re-authenticate." A 403 means "you don't have permission for this resource" — re-authenticating won't help. The old code returned the 401 message for both, which would cause an AI agent to re-authenticate in a loop when the real problem is insufficient permissions. Getting this distinction right is particularly important for the hosted server, where ADR-010 delegates authorization to the CAG layer.

The fix in both test files is identical: change UnauthorizedException to HttpResponseException in the 403 test. This makes the test throw what production actually throws, which routes through handleHttpResponseException instead of the UnauthorizedException catch block, ultimately calling mapHttpErrorCode(403) and returning the correct "Access denied" message.

Tooling: Ralph Agent Supervisor

This branch also carries two new scripts — ralph and ralph-tail — that are unrelated to the error handling change. They're operational tooling for the AIML-110 implementation workflow: ralph is a Bash supervisor that wires br (beads CLI) and Codex into an automated implementation loop, and ralph-tail streams agent messages from the latest run directory. These are committed on the branch because they were developed in parallel and share the same base.

Agent Automation Scripts Operational Tooling scripts/ralph (452 lines), scripts/ralph-tail (60 lines)
ralph takes a parent bead ID and max iterations, then loops: resumes incomplete work, selects a ready bead via br ready, validates eligibility (type, status, repo label, content), dispatches a Codex worker, and saves all artifacts under history/ralph-runs/. ralph-tail is a companion that streams agent messages from the latest run with auto-switching when a newer run appears. Both are standalone Bash scripts with no test coverage (they're developer tooling, not production code). Design doc: plans/AIML-110/ralph-design.md.
