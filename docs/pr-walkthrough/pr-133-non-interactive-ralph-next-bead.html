Context

AIML-110 is building a Contrast-hosted remote MCP server, and the work is organized into numbered slices — each slice stacks on the one before it. The implementation spans two repositories: mcp-contrast (the public stdio MCP server and control plane) and aiml-services (the hosted remote server). An AI agent named Ralph implements beads (granular work items) autonomously, while a companion script called ralph-next-bead handles the transitions between slices: creating pull requests, generating reviewer walkthroughs, and setting up the next stacked branch.

This PR is the mcp-contrast side of Slice 11 (AIML-765), which validates multi-tenant isolation and load testing against the staging environment. The actual isolation tests, load harness, and capacity telemetry were built in aiml-services across three implementation beads. Once those beads closed, ralph-next-bead was invoked to create PRs and walkthroughs for both repos and to set up the Slice 12 branch. That invocation revealed a bug: the skills it calls have interactive decision points that stall when no human is present.

2

The Problem

Picture ralph-next-bead finishing a slice at 2 AM. It calls claude -p "/pr-tools:create-pr ..." to create the pull request. The create-pr skill detects a stacked branch and asks: "Which branch should I target?" Nobody answers. The process hangs indefinitely. The walkthrough skill does the same — its instructions say to ask the user for additional context before writing. The stacked-branch skill asks which branch to base off. Three interactive prompts, three potential deadlocks, and no human in the loop.

The root cause is a mismatch between the skill design and the invocation context. These skills were written for interactive use, where a developer is actively chatting. But ralph-next-bead invokes them via claude -p (print mode), which runs Claude Code non-interactively. The skills don't know they're in automation unless the prompt explicitly tells them.

Why this matters Without this fix, the bead-transition pipeline cannot run unattended. Each transition between slices requires creating PRs and walkthroughs in both repos, then setting up the next branch. If any of the six claude -p invocations (3 skills × 2 repos) stalls, the entire pipeline stops. This blocks Ralph from starting the next slice's work, which in turn delays the overall AIML-110 timeline.
3

Approach

The fix is surgical: append a non-interactive instruction to each prompt string. Rather than modifying the skills themselves (which serve interactive users too), the prompts tell each skill to detect context from the bead and codebase and make decisions autonomously. Each of the three claude -p calls gets a tailored instruction:

create-pr — told to auto-detect the base branch from stacked-pr-facts rather than asking the user to pick one. This is reliable because ralph-next-bead always runs on branches that already have stacked-PR config recorded in local git config.

create-html-pr-walkthrough — told to use bead, PRD, and codebase context to produce the walkthrough in a single pass, skipping the "Is there anything else?" prompt. The walkthrough skill's instructions explicitly say to ask the user for additional context; the prompt override bypasses this.

create-stacked-branch — told to make all decisions autonomously. The prompt already specifies the bead ID and stacking direction, so there's nothing left to decide interactively.

Design choice Prompt-side override, not skill-side change. Modifying the skills to auto-detect non-interactive mode would be cleaner long-term, but it would require changes to the shared pr-tools plugin that serves all projects. The prompt-side approach is local, low-risk, and specific to Ralph's invocation pattern.

Slice 11 in Context

Slice 11 is the "prove it works under load" gate for the hosted MCP server. Before expanding to full tool parity (Slice 12), the team needed confidence that the multi-tenant architecture actually isolates data between organizations under concurrent requests and meets performance SLOs. The work was split across two repos:

mcp-contrast (this repo & PR)
ralph-next-bead non-interactive fix — unblock automation pipeline for slice transitions
PR walkthrough — this document, generated for code reviewers
aiml-services (companion repo)
mcp-uczf — concurrent two-user/two-org isolation integration tests
mcp-8ar6 — representative stateless load and timeout SLO tests
mcp-4wsw — observe-first capacity telemetry integration
mcp-ihab.1 — human-run evidence pass (accepted same-user/two-org)

The heavy lifting happened in aiml-services. There, three implementation beads built concurrent isolation tests proving zero cross-org data leakage, a load harness validating p99 < 30s with no execution exceeding 55s, and Micrometer-based capacity telemetry for response size, fan-out, downstream duration, and auth outcomes. The human operator then ran the final evidence pass and accepted same-user/two-org results as sufficient for forward progress.

This PR's contribution is operational: ensuring the automation pipeline that creates PRs and transitions between slices works without human babysitting. It's a small change with outsized impact on development velocity — every future slice transition runs through these same three claude -p calls.

Summary & What's Next

Three prompt strings, three appended instructions, zero behavioral regression. The skills still work identically for interactive users — the non-interactive directive is part of the automation prompt, not the skill definition. The fix was discovered during the Slice 10→11 transition and applied for all future transitions.
