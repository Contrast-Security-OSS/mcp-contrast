Why This PR Exists

Slice 10 was the first time real staging infrastructure met real MCP clients. For the first time in the AIML-110 project, browser-based OAuth login, bearer token relay through the Contrast API Gateway (CAG), and a matrix of seven different MCP clients were tested against a live staging environment. The bulk of that validation work — the OAuth gates, the Spring AI upgrade, the auth-on-initialize fix, and the client compatibility scripts — all happened in the aiml-services repo. This public-repo PR is the feedback loop: the operational learnings that surfaced during that staging validation and looped back into the shared tooling.

Three concrete problems emerged during client matrix testing. First, Gemini CLI and opencode left local config artifacts (.gemini/ and opencode.json) in the working directory. Second, Ralph attempted to pick up beads labeled ready-for-human that required manual browser-based OAuth testing — work an AI agent cannot perform. Third, ralph-next-bead silently skipped PR creation for repos on stacked branches when the parent branch had not yet been pushed to origin. Each of these problems is addressed by one of the three commits in this PR.

Slice 10 Validation Architecture
Key insight The four working clients (Claude Code, Codex, Copilot CLI, opencode) and two blocked clients (Gemini CLI, VS Code Copilot) collectively surfaced every issue addressed in this PR: config file pollution, Ralph attempting human-only beads, and stacked-branch detection failures.

The client matrix results drove the specific fixes. During S10D certification, seven MCP clients were tested against the staging hosted MCP server. Four passed end-to-end: Claude Code 2.1.150, Codex 0.132.0, Copilot CLI 1.0.51 (after the auth-on-initialize fix in aiml-services), and opencode 1.15.10 (also after that fix). Gemini CLI 0.43.0 was blocked by a DCR empty-scope bug, VS Code Copilot by a DCR rejection, and Claude Desktop remains untested due to its HTTPS requirement. The artifacts left by Gemini CLI and opencode during testing led directly to the .gitignore update, while the human-only S10D testing beads exposed the Ralph filtering gap.

Ralph Learns to Skip Human-Only Beads

Ralph cannot open a browser, but S10D had beads that required exactly that. During Slice 10 validation, several sub-beads — like the client matrix certification bead mcp-g4ot — required manual browser-based OAuth testing. These beads were labeled ready-for-human or needs-human-review to signal that they need a human at the keyboard. Before this fix, Ralph would pick up those beads in its work loop, attempt to dispatch Codex on them, and either fail confusingly or produce meaningless results. The fix introduces a three-layer defense: filter at selection time, filter at resume time, and a hard guard at validation time.

Fixing Stacked-Branch Commit Detection

On stacked branches, the old repo_has_commits could silently report "no commits" and skip PR creation entirely. The ralph-next-bead script orchestrates the transition between beads: creating PRs for completed work, generating walkthroughs, and setting up stacked branches for the next bead. Its repo_has_commits function determines whether a repo has work to PR. The old implementation relied on stacked-pr-facts.sh detect-base-candidates to find the base branch, then counted commits from origin/<base>..HEAD. The problem: on stacked branches where the parent has not yet been pushed to origin, stacked-pr-facts.sh fails and the function returned false — causing the entire PR creation + walkthrough pipeline to be skipped for a repo with real commits.

The fix replaces the single-shot approach with a multi-level fallback cascade. Each level degrades gracefully to the next, and diagnostic variables (_RHC_BASE, _RHC_COUNT, _RHC_METHOD) are exported at every step so the calling code can log exactly which detection strategy was used and what it found.

 # --- Diagnostic variables exported for logging ---

99+_RHC_BASE=""

Diagnostic tracing. Three module-level variables that repo_has_commits populates on every invocation. _RHC_BASE records which base branch was used, _RHC_COUNT holds the commit count, and _RHC_METHOD identifies which detection strategy succeeded. The calling code in process_repo logs all three, making it trivial to diagnose why a repo was processed or skipped.
scripts/ralph-next-bead MODIFIED 
 # --- repo_has_commits: multi-level fallback rewrite ---
 
103+repo_has_commits() {
104+  local repo_dir="$1"
105+  local current_branch default_branch
106+  _RHC_BASE="" ; _RHC_COUNT="" ; _RHC_METHOD=""
107+
108+  current_branch=$(cd "$repo_dir" && git branch --show-current)
109+  default_branch=$(cd "$repo_dir" && git symbolic-ref --short refs/remotes/origin/HEAD 2>/dev/null \
110+    | sed 's#^origin/##' || echo "main")
111+
112+  if [[ "$current_branch" == "$default_branch" ]]; then
113+    _RHC_METHOD="on-default-branch"
114+    return 1
115+  fi
Early exit for default branch. If the repo is on main (or whatever the default branch is), there is no feature work to PR. The function returns false immediately with _RHC_METHOD="on-default-branch".

151+  else

153+    _RHC_METHOD="all-methods-failed"
154+  fi
155+
156+  _RHC_COUNT="$count"
157+  [[ "$count" -gt 0 ]]
158+}
The commit-counting cascade. Level 2 tries origin/<base_branch>..HEAD. If origin/<base_branch> does not exist (the stacked parent has not been pushed yet), Level 3 falls back to origin/<default_branch>..HEAD, which will overcount (including the parent’s commits) but correctly reports "yes, there are commits to PR." Level 4 is the last resort: git log HEAD --not --remotes counts all commits that have not been pushed to any remote. Only if every method fails does the count fall to zero. This cascade is what prevents the silent skip that plagued the old implementation.
