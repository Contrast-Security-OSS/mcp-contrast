Context & Vision

Enterprise customers need a hosted MCP server with OAuth and multi-tenant isolation, and the primary driver is DBS Bank. DBS cannot distribute API keys to developer workstations and operates under strict network lockdown policies. The current MCP server runs locally via stdio transport, which requires every developer machine to hold Contrast credentials. A hosted HTTP deployment mode solves this by centralizing credentials on the server and authenticating users through OAuth.

This PR covers the public repository half of the extraction. All 12 MCP tools move from the monolithic contrast-mcp-stdio-app into a new contrast-mcp-core shared library. The hosted server lives in a separate private repository (aiml-services) and depends on this same core module. Together, these two codebases deliver both deployment modes from a single set of tool implementations.

Why this matters Without the shared core, every tool would need to be duplicated across the stdio and hosted servers. That doubles the maintenance burden, creates divergence risk, and makes it impossible to guarantee that both deployment modes behave identically. The extraction ensures one implementation, two transports.

Architecture Before & After

The monolith becomes a multi-module Gradle project with a clean dependency boundary. Before this PR, every tool class, base class, SDK helper, and data model lived in a single src/ tree. The tools called ContrastSDK and SDKExtension directly, making them impossible to reuse in another runtime. After the extraction, tools depend only on the ContrastApiClient interface, and the concrete SDK wiring lives solely in the stdio app module.

The McpContrastApplication wiring class is now the only file in the stdio module that references all 12 tools. Its only change was replacing a fully-qualified ObjectProvider import with a proper import statement. All tool imports now resolve from the contrast-mcp-core packages. The settings.gradle declares both subprojects:

settings.gradle new

Auth-Agnostic Base Classes

The base tool classes gained a pluggable authentication strategy and tightened their error messages to prevent leaking SDK internals. BaseTool is now a new abstract class in the core module that owns the AuthenticationStrategy injection point and the mapHttpErrorCode() helper. Both PaginatedTool and SingleTool were moved into core with parallel changes: the executePipeline methods gained ToolContext overloads, and exception handling was hardened.

Cursor Pagination

Some Contrast APIs (particularly the dataplatform endpoints used by the hosted server) use keyset/cursor pagination instead of offset pagination. The existing PaginatedTool only supports page/offset. The new CursorPaginatedTool base class provides the same template-method pipeline but uses an opaque cursor token for continuation. No tools in this PR use cursor pagination yet; this is infrastructure for upcoming hosted-server tools.

Security Cursor values are opaque and never logged. CursorPaginationParams.cursorPresence() returns "present" or "absent" for metrics and log entries, never the actual cursor value.

Error Handling & Security

Several layers of defense prevent sensitive data from reaching AI consumers or appearing in logs. This PR introduces HttpRequestRedactor (strips auth headers from vulnerability event data), hardens mapHttpErrorCode messages, adds SdkApiClient.normalizeForbidden() to rewrite SDK 403 errors, and fixes LibraryExtended vulnerability count bugs.

Vulnerability events include the raw HTTP request that triggered the finding. This may contain Authorization: Bearer ... headers. The redactor strips these entirely (line and any folded continuations) before the data is returned to the AI consumer.
SdkApiClient.java (normalizeForbidden) new 
+  private static HttpResponseException normalizeForbidden(UnauthorizedException e) {
+    if (e.getCode() != 403) return e;
+    return new HttpResponseException(
+        "Downstream request failed", "POST", "[redacted]",
+        e.getCode(), e.getStatus(), "[redacted]");
+  }
The SDK's UnauthorizedException for 403 contains the full request URL and response body from the Contrast API. normalizeForbidden() wraps it in a clean HttpResponseException with redacted fields, so even if the exception message leaks upstream, no credentials or internal URLs are exposed.

Test Restructuring

Tests are split into three layers: core unit tests (new), local parity tests (renamed), and integration tests (unchanged). Core unit tests mock ContrastApiClient directly, verifying tool logic in isolation from any transport. Local parity tests mock at the SDK HTTP boundary through SdkApiClient, verifying that the stdio app produces the same results as a direct API call. Integration tests remain unchanged and test against the real Contrast API.

CoreBoundaryTest is an architectural guard that prevents accidental boundary violations. It scans the core module's source tree and asserts: all 12 expected tool files exist, no forbidden types (e.g., McpContrastApplication, ContrastSDK, SdkApiClient) appear in core source, no forbidden text patterns (e.g., @Configuration, SDKHelper) appear in core Java files, and the exact count of 12 production tools matches.

Build & Publication Pipeline

The core library publishes to Contrast's internal Artifactory through a two-job GitHub Actions workflow with a protected environment gate. Artifactory URLs come from Gradle properties (never hardcoded), and the PublishToMavenRepository task rejects execution when the URL property is missing. A comprehensive metadata verification task validates both POM and Gradle module metadata to prevent accidentally publishing Lombok or Spring AI transport dependencies.

Build changes The root build.gradle applies -parameters to all subproject Java compilation (required for Spring AI parameter name resolution). BOM handling is differentiated: the core module uses platform() (non-enforced, consumers can override), while the stdio app uses enforcedPlatform() (locked versions).
Workflow alignment verification scripts (hack/verify-*) STANDARD PATTERN
Shell scripts in hack/ enforce structural invariants during CI:

verify-public-workflow-alignment.sh: Asserts that public CI workflows, Makefile targets, and Dockerfile match expected patterns after the multi-module restructuring.

verify-s8-*.sh (slice 8 gates): Assert that tools live in core (not stdio), tool names are preserved, client abstraction is used (ContrastApiClient not SDK), and test output contains no secrets.
